Formation Git

La gestion de version

C'est la gestion du changement

  • Code source
  • Documentation
  • Configuration
  • Tests
  • Media

Ne concerne pas

  • Les backups
  • Gros fichiers binaires
  • Fichiers auto-générés (processus de build, cache)
  • Tout fichier dont on ne souhaite pas suivre les révisions

Pour travailler seul

  • Conserver un historique
  • Revenir en arrière
  • Ne pas perdre de données
  • Savoir où l'on en est

Pour travailler à plusieurs

  • Partager le travail
  • Éviter les conflits
  • Résoudre les conflits
  • Savoir qui a fait quoi

    • Fonctionnalités avancées

  • Par fichier / projet
  • Gérer des versions parallèles
  • Tagger certaines révisions
  • Réordonner l'historique

    • Types de gestions de versions

    La préhistoire : la gestion manuelle

    Création d'archives, copie dans un répertoire, etc.

    Le moyen-âge : la gestion centralisée

    La version de référence est stockée sur un serveur.

    L'époque moderne : la gestion décentralisée

    Chaque développeur dispose d'une copie sur son environnement.

    Concepts

    Source Code Management (SCM) system

    Logiciel de gestion de version.

    Révision

    Version donnée d'un fichier ou d'un projet.

    Commit

    Action d'enregistrer une nouvelle révision

    Patch

    Fichier qui contient les modifications entre plusieurs révisions.

    Working copy

    Copie locale du projet.

    Branche

    Suite cohérente de révisions.

    Fusion

    Action qui consiste à intégrer dans une même révision les modifications issues de plusieurs branches.

    Tag

    Nom associé à une révision donnée.

    Tip ou HEAD

    Révision la plus récente.

    Les bonnes pratiques

    Commiter atomiquement

    Un commit concerne une seule modification.

    (bug, faute d'orthographe, nouvelle fonctionnalité, refactoring, etc.)

    Être précis et descriptif

    Dans ses messages de commit.

    Commiter souvent

    Dès qu'on a quelque chose à commiter, on le fait.

    Travailler sur des branches

    Pour ne pas casser la branche principale.

    Créer un dépôt par projet

    Au lieu d'un gros dépôt fourre-tout

    Ne pas commiter d'information sensible

    Pour des raisons évidentes
